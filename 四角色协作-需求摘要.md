# 四角色多 Agent 协作 — 需求摘要

> 产品经理与需求方已对齐。本文档为需求基线，供技术架构师出方案、开发经理排期、测试总监评估可测性及四方对齐使用。

---

## 1. 目标与范围

- **产品名称**：基于 OpenClaw 的多 Agent 协作版本（四角色协作）
- **核心目标**：实现「产品经理、技术架构师、开发经理、测试总监」四角色按既定流程的多 Agent 协作，流程可完整跑通（从需求到发布决策/回流）。
- **运行形态**：同一套流程与角色逻辑，在 **Cursor** 与 **OpenClaw（含飞书）** 均可运行；下达指令后达到的效果一致，不区分入口形态。
- **共享核心**：流程与角色逻辑落在 **独立于 OpenClaw 的共享核心**（单独 Git 仓库，当前本地维护，后续上 GitHub）；OpenClaw 与 Cursor 均引用该核心。引用方式由技术方案决定。

---

## 2. 流程（必须支持）

```
需求/优先级(产品) → 方案(架构) → 排期(开发) → 可测性(测试)
                              ↓
                    四方对齐：范围·方案·时间·质量
                              ↓
              开发实现 ←→ 测试执行（架构关键节点把关）
                              ↓
            【产品经理最终验收需求】  ← 按验收标准逐项确认
                    ↓ 通过              ↓ 不通过
              发布决策(四方)            回流开发/测试
```

- 「开发实现 ↔ 测试执行」可多轮交替（提测/修复）。
- 验收不通过时回流到开发/测试，再回到产品验收。

---

## 3. 需求侧已确认的约定

| 项 | 结论 |
|----|------|
| 指令内容 | 支持两种：仅一句话/一段话；或带附件（文档、链接、文件）。 |
| 产出呈现 | 两边效果一致即可，具体展示形式由实现方按 Cursor/飞书载体适配。 |
| 暂停/续跑 | 第一版不需要；支持一口气跑完即可。 |
| 执行层范围 | **第一版做完整执行层**：开发实现可生成/修改代码、触发测试；测试执行可跑真实测试；并对接部署/CI、外部仓库等。需建设完整的「执行层对接」模块。 |

---

## 4. 核心功能模块（实现视角）

| 模块 | 职责 |
|------|------|
| 流程引擎 | 维护阶段与流转（需求→方案→排期→可测性→四方对齐→开发↔测试→产品验收→发布/回流）。 |
| 角色/Agent 定义与调度 | 定义并调度四角色，决定当前该谁执行。 |
| 上下文/共享状态 | 在各角色间存储、传递、读取产出。 |
| 指令解析与入口 | 接收用户指令（文本+可选附件），解析后作为流程输入。 |
| 编排/回合控制 | 按流程决定下一步角色与传入上下文，并执行调用。 |
| 输出呈现与渠道适配 | 按 Cursor/飞书等渠道做展示适配，保证效果一致。 |
| **执行层对接** | 对接代码库、执行环境、测试 runner、部署/CI、外部仓库等；第一版按完整执行层建设。 |

**需产品先确定的模块**：执行层对接 — 已确定为 **第一版做完整执行层**。

---

## 5. 验收标准（产品经理最终验收用）

| 编号 | 验收项 | 通过标准 |
|------|--------|----------|
| A1 | 四角色可区分运行 | 能明确识别当前是哪个角色在输出。 |
| A2 | 流程阶段可识别 | 能识别当前处于流程中哪一阶段。 |
| A3 | 流程可完整走通 | 从「需求/优先级」到「发布决策」或「回流开发/测试」能按顺序走通至少一次。 |
| A4 | 产品验收步骤存在 | 存在产品经理按验收标准做最终验收并给出通过/不通过结论的步骤。 |
| A5 | 回流可触发 | 验收不通过时可进入回流开发/测试并再次进入产品验收。 |
| A6 | 执行层可用 | 开发实现、测试执行能驱动真实环境（代码、测试、与部署/CI或外部仓库的对接按约定可用）。 |
