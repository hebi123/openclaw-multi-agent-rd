## 二、产品经理答复

### 对技术架构师的答复

| 编号 | 答复 |
|------|------|
| A1 | **边界**：共享核心包含「流程定义 + 四 Agent 定义 + 编排逻辑 + 上下文/状态结构 + 执行层接口契约」。具体「谁调 CI、谁写工作区文件」的实现放在**运行侧**（Cursor 或 OpenClaw），通过注入满足核心定义的执行层适配器与核心交互。 |
| A2 | **是**。核心定义执行层**接口/契约**（如：运行测试、提交代码、触发部署、拉取/推送仓库等）；具体对接哪个 CI、哪个仓库由 Cursor 或 OpenClaw 在运行时注入实现，核心不写死具体厂商或仓库地址。 |
| A3 | **需要显式约定**。在完整需求文档中补充：在「开发实现」「测试执行」中，至少各有一个**关键节点**需经架构师确认后才能进入下一阶段（如：开发完成一轮实现后、测试完成一轮用例执行后）。具体节点名与输入输出在架构方案中细化。 |
| A4 | **第一版**：流程状态可仅内存持有，跑完即丢。若有「运行日志/过程产物落盘」需求，视为**可选增强**：若能低成本实现（如默认写一份运行摘要到工作区某路径），第一版可做；若成本高则放到后续版本。由架构师评估后在与开发/测试对齐时定。 |

### 对开发经理的答复

| 编号 | 答复 |

## 三、完整需求文档（PRD）

### 1. 概述

#### 1.1 产品名称与目标

- **产品名称**：基于 OpenClaw 的四 Agent 多 Agent 协作（以下简称「四 Agent 协作」）。
- **核心目标**：在 Cursor 与 OpenClaw（含飞书）中，通过同一套共享核心，实现「产品经理、技术架构师、开发经理、测试总监」四 Agent 按既定流程的多 Agent 协作，流程可完整跑通（从需求到发布决策或回流），且第一版包含完整执行层（代码、测试、部署/CI、外部仓库对接）。

#### 1.2 用户与运行形态

- **用户**：使用 OpenClaw 或 Cursor 的团队或个人。
- **运行形态**：同一套流程与角色逻辑，在 Cursor 与 OpenClaw（含飞书）均可运行；下达指令后达到的效果一致，不区分入口形态。入口与展示形式由各运行侧适配。

#### 1.3 共享核心与仓库

- 流程与角色逻辑落在**独立于 OpenClaw 的共享核心**，单独 Git 仓库，当前本地维护，后续上 GitHub。
- OpenClaw 与 Cursor 均引用该核心；引用方式由技术方案决定。
- **核心包含**：流程定义、四 Agent 定义、编排逻辑、上下文/状态结构、**执行层接口契约**。
- **核心不包含**：具体 CI/仓库/工作区的实现；具体实现由各运行侧注入适配器满足核心契约。

---

### 2. 角色与流程

#### 2.1 四 Agent 定义

| 角色 | 职责焦点 | 主要产出 |
|------|----------|----------|
| 产品经理 | 需求、优先级、验收标准 | 需求/PRD、验收标准、最终验收结论（通过/不通过） |
| 技术架构师 | 方案、技术选型、关键节点把关 | 架构方案、接口/约束、关键节点确认意见 |
| 开发经理 | 排期、实现、代码与交付 | 排期计划、任务拆解、实现产物、提交/部署相关结果 |
| 测试总监 | 可测性、测试策略与执行、质量结论 | 可测性结论、测试策略/用例、测试执行结果、发布建议 |

#### 2.2 流程定义（必须支持）

```
需求/优先级(产品) → 方案(架构) → 排期(开发) → 可测性(测试)
                              ↓
                    四方对齐：范围·方案·时间·质量
                              ↓
              开发实现 ←→ 测试执行（架构关键节点把关）
                              ↓
            【产品经理最终验收需求】  ← 按验收标准逐项确认
                    ↓ 通过              ↓ 不通过
              发布决策(四方)            回流开发/测试（最多 1 次）
```

- 「开发实现 ↔ 测试执行」可多轮交替（提测/修复）。
- **架构关键节点把关**：在「开发实现」「测试执行」中，至少各有一个关键节点需经架构师确认后才能进入下一阶段；具体节点与输入输出在架构方案中细化。
- **回流**：验收不通过时，可回流开发/测试**最多 1 次**；若再次验收仍不通过，流程结束并输出「未通过」结论。

#### 2.3 指令与附件

- 支持两种指令形式：**仅文本**（一句话或一段话）；**文本 + 附件**（文档、链接、文件）。
- 附件在全流程中作为**共享上下文**可供各角色使用；具体解析与传递方式由实现方决定。

#### 2.4 产出呈现与一致性

- Cursor 与飞书侧：**关键产出内容一致**（需求摘要、方案要点、排期、可测性、验收结论、发布建议等）；展示格式可因载体不同而不同。
- 第一版**不支持**暂停/续跑；流程状态可仅内存持有，跑完即丢。运行日志/过程产物落盘为可选增强，由架构评估后与开发/测试对齐决定是否纳入第一版。

### 3. 功能需求

#### 3.1 功能模块与职责

| 模块 | 职责 | 备注 |
|------|------|------|
| 流程引擎 | 维护当前阶段及流转；支持回流且最多 1 次。 | 阶段定义见 2.2 |
| 角色/Agent 定义与调度 | 定义四 Agent 能力与调用方式；在流程中决定当前该谁执行。 | |
| 上下文/共享状态 | 在各角色间存储、传递、读取产出与附件等。 | 结构由架构方案定义 |
| 指令解析与入口 | 接收用户指令（文本+可选附件），解析后作为流程输入。 | 入口形态不限定 |
| 编排/回合控制 | 按流程决定下一步角色与传入上下文，并执行调用。 | |
| 输出呈现与渠道适配 | 按 Cursor/飞书等做展示适配，保证关键产出内容一致。 | |
| 执行层对接 | 满足核心定义的**执行层接口契约**：代码读写、测试运行、提交、部署/CI、外部仓库等；具体实现由运行侧注入。 | 见 3.2 |

#### 3.2 执行层对接（第一版完整执行层）

- **核心侧**：定义并持有**执行层接口/契约**（如：运行测试、提交代码、触发部署、拉取/推送仓库等），不绑定具体 CI 或仓库。
- **运行侧**：Cursor 或 OpenClaw 注入具体实现（对接具体 CI、仓库、工作区），满足契约。
- **约定**：由架构师与开发输出《执行层对接约定》初版，明确支持的 CI/仓库类型、必选与可选能力；产品评审后作为 A6 验收依据。执行层相关测试使用独立测试仓库/CI 或 mock，不依赖用户真实生产环境。

#### 3.3 开发顺序建议（已与产品确认）

1. 流程引擎  
2. 上下文/共享状态  
3. 角色定义与调度  
4. 编排/回合控制  
5. 指令解析与入口  
6. 输出呈现与渠道适配  
7. 执行层对接

### 4. 非功能需求

| 项 | 要求 |
|----|------|
| 一致性 | 同一指令在 Cursor 与在 OpenClaw（飞书）下，关键产出内容一致。 |
| 可测试性 | 流程可完整走通具备可重复 E2E 用例；执行层测试使用独立环境或 mock。 |
| 可维护性 | 流程与角色在共享核心内单一维护，运行侧仅做接入与适配。 |

---

### 5. 验收标准（产品经理最终验收）

| 编号 | 验收项 | 通过标准 |
|------|--------|----------|
| A1 | 四 Agent 可区分运行 | 能明确识别当前是哪个 Agent 在输出。 |
| A2 | 流程阶段可识别 | 能识别当前处于流程中哪一阶段。 |
| A3 | 流程可完整走通 | 从「需求/优先级」到「发布决策」或「回流开发/测试」能按顺序走通至少一次；具备可重复 E2E 用例。 |
| A4 | 产品验收步骤存在 | 存在产品经理按验收标准做最终验收并给出通过/不通过结论的步骤。 |
| A5 | 回流可触发且受限 | 验收不通过时可回流开发/测试；最多 1 次回流，再次不通过则流程结束并输出「未通过」。 |
| A6 | 执行层可用 | 开发实现、测试执行能驱动真实环境；与部署/CI 或外部仓库的对接符合《执行层对接约定》。 |

---

### 6. 约束与假设

- **回流**：第一版回流次数上限为 1 次。
- **暂停/续跑**：第一版不支持；状态可仅内存，跑完即丢；落盘为可选。
- **执行层**：核心只定义契约；具体 CI/仓库由运行侧注入；测试不依赖生产环境。
- **《执行层对接约定》**：由架构与开发输出、产品评审后作为 A6 依据。
