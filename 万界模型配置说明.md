# 万界方舟（CC）模型配置说明

参考文档：[万界方舟接口文档](https://docs.wjark.com/maas/Interface.html)

## 已按文档修正的配置

- **Base Url**：已改为万界 **OpenAI 兼容** 地址 `https://maas-openapi.wanjiedata.com/api`（与文档一致）。
- **调用方式**：已改为 `openai-completions`，请求会发往 `POST /v1/chat/completions`，鉴权为 **Authorization: Bearer &lt;API KEY&gt;**，与文档示例一致。
- **API Key**：使用 `env.WANJIE_API_KEY`（在 [万界方舟 API Key](https://www.wjark.com/center/api-key) 复制）。

## 模型 ID 必须与「模型广场」一致

文档要求：**授权模型名称需在「模型广场」中点击模型名称进行复制。**

若仍报 425 或模型不存在：

1. 打开 [万界方舟-模型平台](https://www.wjark.com/)，登录后进入 **模型广场**。
2. 找到你要用的 Claude Opus 4.5 模型，**点击名称复制** 得到准确 ID。
3. 在 `openclaw.json` 的 `models.providers.anthropic.models[0].id` 和 `agents.defaults.model.primary` 中，把 `claude-opus-4-5-20251101` 替换为复制的 ID（若不同）。

## 425「密钥未激活」

若接口与模型 ID 都正确仍返回 425，表示该 API Key 在万界侧未激活，需：

- 在万界后台查看该 Key 的激活/开通状态，或  
- 按万界要求运行其提供的激活脚本并完成绑定/激活后再试。

## 万界 + 飞书时的兼容行为（避免 404）

万界接口对 OpenAI SDK 的请求会返回 404，因此对万界做了单独处理（在 pi-ai 的 openai-completions 中）：

- **万界走裸 fetch**：当 `provider=wanjie` 或 baseUrl 包含 `wanjiedata.com` 时，不再使用 OpenAI SDK，改为用 `fetch` 直连 `POST /v1/chat/completions`，仅带 `Authorization` 与 `Content-Type: application/json`，请求体与文档一致，从而避免 404。
- **完整能力 + 省 token**：对万界使用与其它厂商一致的请求体（含 `tools`、`tool_choice`、`store`、`stream_options`、`max_completion_tokens` 等），但**每条请求只带最近 16 条消息**（首条 + 最近 15 条），以控制单次请求 token，避免很快触发万界每日费用上限。

修改后需**重启网关**生效：`openclaw gateway stop` → `openclaw gateway`。

## 飞书/渠道里看到「HTTP 404: 404 page not found」

说明调用万界接口时收到了 HTTP 404，错误被当成助手回复发到了渠道。请按下面排查：

1. **确认 baseUrl**：`openclaw.json` 里 `models.providers.anthropic.baseUrl` 应为 `https://maas-openapi.wanjiedata.com/api`（不要漏掉 `/api`，也不要多加斜杠）。
2. **确认模型 ID**：与「模型广场」中复制的名称完全一致（见上文「模型 ID 必须与「模型广场」一致」）。
3. **确认 API Key**：环境变量 `WANJIE_API_KEY` 或配置里的 apiKey 已正确设置，且在万界后台已激活。
4. **本地用 curl 自测**：
   ```bash
   export API_KEY="你的API_KEY"
   curl -sS -X POST 'https://maas-openapi.wanjiedata.com/api/v1/chat/completions' \
     -H "Authorization: Bearer $API_KEY" \
     -H "Content-Type: application/json" \
     -d '{"model":"你的模型ID","messages":[{"role":"user","content":"hi"}],"stream":false}'
   ```
   若这里就返回 404，多半是模型 ID 或 Key 问题；若这里正常而 OpenClaw 仍 404，再看网关日志里模型请求的完整 URL 和错误信息。

修复后重启网关：`openclaw gateway stop` → `openclaw gateway`。  
当前版本会在检测到此类原始 HTTP 错误时，在飞书里替换为友好提示，并在网关日志中打印原始错误，便于排查。

## 仍 404 时：抓「实际请求 URL + 请求体」并对比 curl

若 baseUrl/模型/Key 都确认无误、且上面 `stream: false` 的 curl 能 200，但飞书里仍 404，可按下面步骤收窄是「哪个字段或头」导致万界返回 404。

### 第一步：在网关里打出实际请求 URL 和请求体

1. **关闭当前网关**（若在运行）：在运行网关的终端按 `Ctrl+C`，或执行 `openclaw gateway stop`。
2. **设置调试环境变量并启动网关**（在同一终端执行）：
   ```bash
   export OPENCLAW_DEBUG_WANJIE=1
   openclaw gateway
   ```
3. **在飞书里给机器人发一条消息**（例如「测试」），触发一次模型请求。
4. **看网关终端输出**，会多出两行类似：
   - `[openclaw-debug] request URL: https://maas-openapi.wanjiedata.com/api/v1/chat/completions`
   - `[openclaw-debug] request body (params): { ... }`
5. **把这两行完整复制保存**（可保存到本地文件或发给我）。请求体里包含 `model`、`stream`、`messages`、以及是否有 `stream_options`、`tools`、`tool_choice` 等，便于和 curl 对比。

### 第二步：用 curl 测「仅 stream: true」是否 404

在**新开一个终端**里执行（替换成你的 Key 和模型 ID）：

```bash
export WANJIE_API_KEY="你的万界API_KEY"
export MODEL_ID="claude-opus-4-5-20251101"

curl -sS -w "\n\nHTTP_CODE:%{http_code}\n" -X POST 'https://maas-openapi.wanjiedata.com/api/v1/chat/completions' \
  -H "Authorization: Bearer $WANJIE_API_KEY" \
  -H "Content-Type: application/json" \
  -d "{\"model\":\"$MODEL_ID\",\"messages\":[{\"role\":\"user\",\"content\":\"hi\"}],\"stream\":true}"
```

- 若输出里有 `HTTP_CODE:200` 且有一串 data 行：说明万界支持 `stream: true`，问题多半在网关请求里**多出来的字段**（如 `stream_options`、`tools`、`tool_choice` 等）。
- 若 `HTTP_CODE:404`：说明仅 `stream: true` 就被 404，需要再对比「非流式」curl（上面 `stream: false`）是否 200，或查万界文档是否流式接口路径不同。

### 第三步：用 curl 尽量复现网关的请求体（可选）

把第一步里日志中的 `request body (params)` 整段复制，保存为 `request-body.json`（确保是合法 JSON）。然后：

```bash
curl -sS -w "\n\nHTTP_CODE:%{http_code}\n" -X POST 'https://maas-openapi.wanjiedata.com/api/v1/chat/completions' \
  -H "Authorization: Bearer $WANJIE_API_KEY" \
  -H "Content-Type: application/json" \
  -d @request-body.json
```

- 若这里也 404：说明是**请求体里某个字段**导致（可逐项删字段重试，例如先删 `stream_options`、再删 `tools`，看删到哪一步变 200）。
- 若这里 200：说明同样请求体在 curl 下正常，则可能是**请求头**不同（如 User-Agent、其他 header）或网关侧实际发出的 URL 与日志不一致，需要再对照网关日志里的 URL 和头。

### 第四步：根据结果收窄

- **第二步 200、第三步 404**：逐项去掉 request-body.json 里的字段（如 `stream_options`、`tools`、`tool_choice`），用 curl 再试，确定是哪个字段触发 404，再在配置或 pi-ai 里为该厂商关闭/改写该字段。
- **第二步就 404**：查万界文档流式接口是否另有路径或参数；或向万界确认是否支持 `stream: true` 及正确用法。
- **第二步和第三步都 200**：对比网关请求头与 curl 的差异（如 User-Agent、Cookie 等），或在网关侧加打印实际发出的 headers 再比对。

## Token 节省建议（避免很快用满每日费用上限）

**每次会话费 token 的主要原因（审计结论）：**

- **系统提示**：bootstrap 文件（AGENTS.md、SOUL.md、TOOLS.md、USER.md、MEMORY.md 等）每轮都会注入，总长受 `bootstrapMaxChars` / `bootstrapTotalMaxChars` 限制；另有固定段落（安全、技能列表、时间、运行时等）。
- **工具定义（主要大户）**：每条请求都会把**当前所有工具的完整 JSON Schema**（name + description + parameters）发给万界，工具越多（含技能暴露的工具）token 越多。
- **对话历史**：万界已限制为「首条 + 最近 15 条」共 16 条，更早的不发。

若希望少烧 token、拉长可用天数，可参考以下做法：

1. **Bootstrap 限制（已写入你当前配置）**  
   在 `openclaw.json` 的 `agents.defaults` 下已设置：
   - `bootstrapMaxChars: 8000`：单份注入文件最多 8000 字符。
   - `bootstrapTotalMaxChars: 24000`：所有注入的 bootstrap 总字符上限 24000。  
   可再按需调小（如 5000 / 15000），系统提示会变短，单次请求 token 更少。

2. **对话历史条数**  
   万界请求已限制为「首条 + 最近 15 条」共 16 条消息；更早的轮次不会发给模型，从而减少每轮 token。

3. **保持工作区文件精简**  
   系统提示会包含 `AGENTS.md`、`SOUL.md`、`TOOLS.md` 等；内容越长，每轮 token 越多。建议只保留必要说明，把详细文档放到外链或 `/context` 之外。`MEMORY.md` 若很大，可把冗长内容迁到 `memory/*.md`，用 memory 工具按需读取，避免每轮全量注入。

4. **减少「工具数量」以降低每轮 token**  
   工具列表（含技能暴露的工具）会以**完整 schema** 每轮下发。可：
   - 在配置里只启用当前渠道需要的技能（技能 allowlist），减少注册工具数；
   - 技能描述尽量简短（会出现在系统提示的技能列表和工具 description 里）。

5. **先看清再优化：/context**  
   在会话中发 `/context list` 或 `/context detail`，可看到当前系统提示各部分的篇幅（bootstrap、工具、技能等），便于针对性裁剪。

6. **善用 /compact**  
   在飞书或 CLI 中可发送 `/compact`，让助手总结旧对话并压缩历史，后续请求只带压缩后的摘要，减少历史 token。

7. **ROLES.md**  
   未列入默认 bootstrap，不会每轮自动注入；仅在处理需求时按 AGENTS.md 要求读取。若内容很长，可保持精简或拆成「简要版 + 按需读详细文档」。

### 按你当前配置的「优先做哪几条」

基于你当前的 `openclaw.json` 和会话里的技能快照（系统提示约 26,796 字符 ≈ 6,700 token，其中 AGENTS.md 单文件 7,804 字符、8 个技能），建议**按下面顺序**做，性价比最高：

| 优先级 | 做什么 | 原因 |
|--------|--------|------|
| **1** | **精简 AGENTS.md** | 当前单文件 7,804 字符，占 Project Context 一大半。保留「Every Session」和必要规则，把 Memory/Safety/Group Chats 等大段改成简短要点或「需要时读 XX.md」。目标：压到约 4,000 字符以内，每轮可少约 1,000 token。 |
| **2** | **关掉飞书用不到的技能** | 当前 8 个技能（4 个 feishu-* + healthcheck、skill-creator、weather、skill-vetting）。飞书日常对话很少用到 healthcheck、skill-creator、skill-vetting、weather。在 `openclaw.json` 里加 `skills.entries`，把不需要的设为 `enabled: false`，例如只保留飞书相关 + 必要时保留 skill-vetting。技能少了 → 工具 schema 和技能列表都变短，每轮 token 明显下降。 |
| **3** | **（可选）再压 bootstrap 上限** | 你已是 8000/24000。若可接受 AGENTS 等被截断，可改为 `bootstrapMaxChars: 5000`、`bootstrapTotalMaxChars: 15000`，进一步压低每轮系统提示。 |
| **4** | **长会话用 /compact** | 飞书里某条会话聊多了，发一次 `/compact` 把旧对话压成摘要，后续就只带摘要 + 最近几轮，避免历史把 16 条额度占满。 |
| **5** | **确认未开 heartbeat** | 你当前配置里没有 `agents.defaults.heartbeat`。若以后加了或默认开启，心跳会定期带完整上下文跑一遍；若主要用飞书、不需要后台检查，可关掉或把间隔设得很长。 |

**技能配置示例**（只保留飞书相关 + 可选 weather，其余关闭）：

```json5
// 在 openclaw.json 的顶层增加或合并 "skills"：
"skills": {
  "entries": {
    "healthcheck": { "enabled": false },
    "skill-creator": { "enabled": false },
    "skill-vetting": { "enabled": false },
    "weather": { "enabled": false }
  }
}
```

这样会话里只会带上 4 个 feishu-* 技能；若需要天气或技能审查，再把对应项改为 `true` 或删掉该条即可。改完后**新会话**生效（当前会话会沿用旧技能快照）。
